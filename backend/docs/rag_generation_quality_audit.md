# Анализ RAG, генерации задач и качества оценки графов

Дата аудита: 2026-06-16.

Окружение проверки:

- Python: `backend/.venv/Scripts/python.exe`
- LLM provider для прогонов: `OpenAI`
- Embedding model: `text-embedding-3-small`
- База: 1783 клинических протокола, 88292 чанка протоколов, 8975 медицинских понятий.

## 1. RAG: скорость

Онлайн RAG состоит из двух основных частей:

1. embedding пользовательского запроса;
2. pgvector-поиск по `protocol_chunks.embedding`;
3. LLM-генерация ответа или графа.

Фактический замер embedding через OpenAI:

| Запрос | Время |
|---|---:|
| первый запрос после старта | 1331 мс |
| последующие короткие запросы | 208-212 мс |
| среднее по 4 запросам | 490 мс |

Фактический замер только vector search по уже сохраненным embedding:

| Метрика | Значение |
|---|---:|
| n | 20 |
| min | 46 мс |
| p50 | 122 мс |
| average | 128 мс |
| max | 217 мс |
| self-retrieval top-1 same chunk | 16/20 |
| self-retrieval top-1 same protocol | 16/20 |

Вывод: сама векторная БД работает приемлемо быстро для текущего объема. Основная задержка в типовом запросе находится в embedding API и LLM-генерации.

## 2. RAG: точность retrieval

В исходном виде retrieval имел два слабых места:

- глобальный dense retrieval мог отдавать нерелевантные протоколы;
- в top-k попадали низкосигнальные чанки: `Внимание!`, `Отправить файл себе на почту`, `Прикрепленные файлы`, base64-изображения, OCR-мусор вида `AAAA...`.

Реализованы улучшения:

- фильтрация низкосигнальных чанков;
- дополнительный title-candidate retrieval по названиям протоколов;
- rerank теперь учитывает vector score, совпадение с заголовком и overlap ключевых слов с текстом чанка.
- section-aware rerank дополнительно покрывает явно запрошенные разделы: диагностика, лечение,
  госпитализация, классификация, мониторинг/наблюдение;
- для title-matched протоколов расширен candidate pool, чтобы короткие, но клинически важные
  разделы не терялись ниже top-k;
- добавлено доменное расширение терминов: например, `гипертензия` учитывает
  `гипотензивная/антигипертензивная терапия`, `лечение` учитывает `терапия`,
  `мониторинг` учитывает `наблюдение/контроль/КТГ`;
- benchmark section-hit использует мягкую нормализацию русских названий разделов, поэтому
  педагогически эквивалентные названия засчитываются как совпадение.

## 3. Генерация клинических сценариев

На протоколе `Артериальная гипертензия у беременных` генерация сценариев дала педагогически пригодные варианты:

- хроническая артериальная гипертензия во время беременности;
- гестационная гипертензия;
- преэклампсия;
- тяжелая преэклампсия;
- послеродовой мониторинг.

Сильная сторона: сценарии клинически похожи на реальные учебные виньетки.

Слабая сторона: без явной рубрики сложности и целевой компетенции сценарии трудно использовать в стандартизированном исследовании.

Реализованное усиление:

- сценарии теперь запрашиваются с полями `difficulty`, `target_competency`,
  `expected_reasoning_steps`, `red_flags`;
- эти поля добавлены в API-схему как optional, поэтому старые клиенты остаются
  совместимыми.

## 4. Генерация эталонного графа

Первый прогон выявил дефект нормализации: LLM иногда помещала категорию узла в поле `type`, а не в `data.category`. Нормализатор заменял такие узлы на `SYMPTOM`, из-за чего граф становился педагогически неверным.

Исправлено:

- если `data.category` отсутствует, нормализатор берет категорию из `node.category` или `node.type`;
- если `node.type` содержит медицинскую категорию, оно заменяется на технический тип `med`.

Второй дефект: LLM иногда генерировала недопустимые типы связей:

- `SUPPORTS`;
- `CONSISTENT_WITH`;
- `INDICATED_FOR` от `DIAGNOSIS` к `EXAM`.

Исправлено:

- неизвестные связи переводятся в `DETERMINES`;
- недопустимая `INDICATED_FOR` от диагноза к обследованию переводится в `REQUIRES_CONFIRMATION`;
- недопустимая `CONTRAINDICATED_DUE_TO` переводится в `DETERMINES`;
- все такие случаи сохраняются в `validation_warnings`.

Контрольный валидный граф после правок:

- 9 узлов;
- 7 ребер;
- warnings отсутствуют в финальном прогоне.

Выявленная педагогическая проблема: онтология не содержала отдельного типа для мониторинга/наблюдения. Поэтому LLM иногда относила `Контроль артериального давления` к `MEDICATION`, хотя это не лекарственное назначение.

Реализованное усиление:

- добавлен `MONITORING` в `NodeType`;
- `INDICATED_FOR` теперь разрешен от `DIAGNOSIS` к `MEDICATION`, `SURGERY` или `MONITORING`;
- в prompt добавлено правило: `MEDICATION` использовать только для препарата, класса препаратов или лекарственной терапии;
- нормализатор уточняет категорию по label: `контроль`, `мониторинг`, `наблюдение`, `КТГ` и похожие формулировки переводятся в `MONITORING`.

Дополнительное усиление 2026-06-17:

- генерация клинических задач, сценариев, проверки текстового ответа и эталонного графа переведена на OpenAI JSON schema output при наличии OpenAI;
- добавлен `graph_generation_judge`: он проверяет наличие `DIAGNOSIS`, диагностического шага, пути от данных пациента к диагнозу, пути от диагноза к лечению, терапию без диагноза, слишком общие узлы и `MONITORING`, ошибочно размеченный как `MEDICATION`;
- API `reference-graph` теперь возвращает `generation_quality`, а предупреждения judge добавляются в `validation_warnings`;
- graph benchmark теперь содержит блок `reference_quality` с метриками `schema_valid_rate`, `accepted_rate`, `warning_rate`, `critical_rate`, `avg_quality_score`.

## 5. Проверка графов и сравнение метрик

На сгенерированном эталоне были построены варианты студенческого решения:

| Вариант | Edge F1 | Weighted Edge F1 | Node Coverage | Category Accuracy | Directed Path | Safety Penalty | Composite |
|---|---:|---:|---:|---:|---:|---:|---:|
| Идеальный граф | 1.00 | 1.00 | 1.00 | 1.00 | 1.00 | 0.00 | 1.00 |
| Пропущены лабораторные узлы | 0.83 | 0.82 | 0.79 | 0.79 | 1.00 | 0.29 | 0.78 |
| Неверная категория симптома | 1.00 | 1.00 | 0.97 | 0.91 | 1.00 | 0.00 | 0.98 |
| Неверная терапевтическая связь | 0.86 | 0.82 | 1.00 | 1.00 | 0.50 | 0.50 | 0.68 |
| Лишнее потенциально опасное лечение до усиления | 0.93 | 0.91 | 1.00 | 1.00 | 1.00 | 0.21 | 0.91 |
| Лишнее потенциально опасное лечение после усиления | 0.93 | 0.91 | 1.00 | 1.00 | 1.00 | 1.00 | 0.52 |
| Разорвана клиническая цепочка | 0.73 | 0.78 | 1.00 | 1.00 | 0.00 | 0.00 | 0.66 |

Вывод: новые метрики дают более честную педагогическую картину, чем один Edge F1.

- Ошибка категории почти не видна в Edge F1, но видна в `category_accuracy`.
- Неверная терапевтическая связь получает сильное снижение через `safety_penalty` и `directed_path_completeness`.
- Разрыв рассуждения явно ловится через `directed_path_completeness`.
- Лишнее опасное лечение после усиления получает `unsafe_extra_action=1.0`,
  `safety_penalty=1.0` и существенное снижение итоговой оценки.

Реализованное усиление:

- вес `safety_penalty` увеличен с `0.25` до `0.35`;
- safety penalty разделен на `missing_critical_action` и `unsafe_extra_action`;
- добавлен hard cap: при лишнем опасном действии итоговая оценка не может быть выше `0.75`;
- `safety_findings` сохраняет объяснимый список опасных ошибок.

Дополнительное усиление graph benchmark 2026-06-17:

- добавлен генератор `scripts/build_graph_eval_seed.py`;
- создан deterministic research seed `benchmarks/graph_cases.research.seed.json`;

## 6. Улучшения для статьи

Для научной части можно описать систему как конвейер:

1. протоколы МЗ РК;
2. semantic chunking;
3. hybrid RAG retrieval;
4. generation of clinical scenario;
5. generation and normalization of reference MedConstructor graph;
6. student graph construction;
7. clinical-weighted graph evaluation.

Для эмпирической части нужны два набора:

- RAG retrieval benchmark: 50-100 вопросов с ожидаемым protocol_id, expected sections и key phrases.
  Первый draft-набор на 50 вопросов уже добавлен как `benchmarks/rag_queries.research.seed.json`;
- graph grading benchmark: 20-30 эталонных графов и набор типовых студенческих ошибок.
  Первый deterministic seed уже добавлен: 20 эталонных графов и 160 вариантов ошибок.

Основные метрики для статьи:

- retrieval: Recall@k, MRR, section hit rate, key phrase hit rate, latency p50/p95;
- generation: schema validity rate, ontology validity rate, warning rate, expert acceptance rate;
- evaluation: correlation with expert grades, sensitivity to critical errors, inter-rater agreement.
  Инфраструктура для этого шага уже добавлена, но нужны реальные оценки преподавателей.
